CONTESTO / COSA È GIÀ PRONTO (lato server + marketplace):
- Item.visibility = "pubblico" | "privato". Gli item privati non compaiono nel
  marketplace (GET /api/items li esclude), non si vendono (prezzo 0), ma l'autore
  li usa nelle sue visite guidate. Creati dall'editor marketplace (checkbox 🔒).
- Visit.accessKey = parola chiave univoca (es. "Fenice rossa"). La sua presenza
  marca la visita come GUIDATA: gratuita, non comprabile, non listata nel
  marketplace dei visitatori. Creata dall'editor marketplace (toggle 🔑).
- Vincolo anti-scappatoia (validato lato server in POST /api/visits): una visita
  guidata può contenere SOLO item gratuiti o posseduti dall'autore.
- Backbone sessioni EFFIMERO in memoria sul server: routes/guidedSessions.ts
  montato su /api/guided-sessions. Nessuna persistenza (a fine visita/riavvio
  sparisce → "nessuna traccia"). Trasporto scelto: POLLING REST (no WebSocket/SSE).

API SESSIONI GIÀ DISPONIBILI (da consumare dal navigator):
  POST /api/guided-sessions            { visitId, teacher }
       -> il DOCENTE apre la sala d'attesa. Ritorna vista docente { id, stato:"attesa", ... }.
  POST /api/guided-sessions/join       { accessKey, username }
       -> lo STUDENTE entra digitando la parola chiave. 404 se chiave inesistente.
          Ritorna { id (sessionId), stato, currentStep, ... }.
  POST /api/guided-sessions/:id/ready  { username, pronto? }   -> studente "sono pronto".
  POST /api/guided-sessions/:id/leave  { username }            -> studente esce.
  POST /api/guided-sessions/:id/start  { teacher }             -> stato "attiva", step 0.
  POST /api/guided-sessions/:id/step   { teacher, index, ritardoMs? }
       -> porta tutti sull'opera `index`; imposta stepStartAt = now + ritardoMs
          (partenza audio ~simultanea nonostante il polling).
  POST /api/guided-sessions/:id/end    { teacher }             -> distrugge la sessione.
  GET  /api/guided-sessions/:id            -> vista DOCENTE: { stato, currentStep, stepStartAt, partecipanti:[{username,pronto}] }
  GET  /api/guided-sessions/:id/state      -> vista STUDENTE: { stato, currentStep, stepStartAt, partecipanti:<numero> }.
       410 se la sessione è terminata (il client deve uscire senza lasciare traccia).
  GET  /api/guided-sessions/:id/items?username=...
       -> contenuti della visita (item con `about` popolato, ordinati) SOLO per un
          partecipante di una sessione viva = accesso TEMPORANEO ai testi (anche
          item privati/a pagamento del docente). 403 se non partecipa, 410 se finita.

DA FARE NEL NAVIGATOR:

1) UI STUDENTE — ingresso con parola chiave.
   [FATTO NEL MARKETPLACE] Il campo "Hai una parola chiave?" è già nella dashboard
   visitatore: chiama POST /api/guided-sessions/join, entra in sala d'attesa e
   mostra un link "Vai alla sala d'attesa" che apre il NAVIGATOR con deep-link:
       <navigator>/?guidedSession=<sessionId>&role=studente&user=<username>
   DA FARE NEL NAVIGATOR: leggere quei query param all'avvio (come già si legge
   ?museum=&visit=), riconoscere role=studente + guidedSession, e mostrare la
   schermata di ATTESA ("in attesa che il docente avvii..."). Lo studente è GIÀ
   nella sala d'attesa lato server (il join è avvenuto nel marketplace); il
   navigator deve solo agganciarsi a quella sessione via il sessionId.
   - Polling GET /api/guided-sessions/:id/state ogni ~2s:
       * stato "attesa"  -> resta in sala d'attesa (può premere "sono pronto"
         -> POST /:id/ready).
       * stato "attiva"  -> entra nella visita. Caricare i contenuti da
         GET /api/guided-sessions/:id/items?username=... (NON da /api/visits/:id/items,
         così l'accesso resta legato alla sessione viva).
       * 410              -> visita terminata: uscire, svuotare guidedSessionId,
         NON lasciare la visita in collezione (possesso temporaneo).

2) SINCRONIZZAZIONE DEI CONTENUTI durante la visita (studente).
   - Durante lo stato "attiva", continuare il polling GET /:id/state ogni ~1s.
   - Quando `currentStep` cambia: portare la vista sull'opera itemListElement[currentStep]
     e programmare la partenza dell'audio (TTS già esistente, useTTS) a `stepStartAt`
     (wall-clock): se stepStartAt è nel futuro, setTimeout(stepStartAt - Date.now());
     se è già passato, partire subito. Lo studente NON può usare "prossimo/precedente"
     (navigazione guidata dal docente) — disabilitare quei comandi in modalità guidata.

3) UI DOCENTE — sala d'attesa e conduzione.
   - Punto d'ingresso: il docente (autore) apre una sua visita guidata e preme "Avvia".
     (Il pulsante può stare nel marketplace e fare un deep-link al navigator, oppure
     nascere nel navigator; la sessione si crea con POST /api/guided-sessions.)
   - Schermata sala d'attesa: polling GET /api/guided-sessions/:id ogni ~2s,
     mostrare la lista partecipanti [{username, pronto}] (chi è collegato / pronto).
   - Pulsante "Avvia visita" -> POST /:id/start (idealmente quando tutti pronti).
   - Durante la visita, avanzando di opera: POST /:id/step { index, ritardoMs: ~500 }
     ad ogni cambio opera (fa partire l'audio ~insieme su tutti).
   - Pulsante "Termina" -> POST /:id/end (la sessione sparisce per tutti).

4) CONFIG / STATE nel navigator.
   - Aggiungere in state.ts i riferimenti: guidedSessionId, guidedRole ("docente"|"studente"),
     e la gestione del possesso TEMPORANEO (non scrivere in User.collezione).
   - api.ts: wrapper fetch per gli 8 endpoint sopra.
